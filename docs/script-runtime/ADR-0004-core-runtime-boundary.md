# ADR-0004: C++ core runtime source boundary

Status: Accepted for taxonomy v5 inventory; implementation and parity remain
separate gates.

## Context

The migration architecture deliberately separates frequently updated game
procedures from stable native components. Python under `module/*` is candidate
script source. Python under `core/*` implements the device, vision, OCR,
configuration, logging, scheduling, resource, and orchestration substrate that
is being replaced by C++.

Taxonomy v4 assigned both trees to `SCRIPT_RUNTIME`. That preserved every call
site, but it made an unresolved implementation detail inside `core/*` look like
a missing script API. It also allowed resolved core helpers to fall through to
`SCRIPT_LANGUAGE_OR_MODULE`, which contradicted the target architecture.

## Decision

Taxonomy v5 assigns source scope before operation ownership:

| Python source | Source scope | Default disposition | Meaning |
| --- | --- | --- | --- |
| `core/*` | `CPP_RUNTIME` | `CPP_RUNTIME_INTERNAL` | Reimplement inside the native runtime; no script ABI is created |
| `module/*`, `main.py`, `cli.example.py` | `SCRIPT_RUNTIME` | no automatic disposition | Resolve to language/module, Host, dependency, or an explicit gap |
| `service/*` and service entry points | `CPP_SERVICE` | `CPP_SERVICE_INTERNAL` | Reimplement inside the native service |

An exact source-and-symbol rule still precedes the source default. This keeps
the previously audited updater/tooling, raw service IPC/listener, task registry,
and narrow Device/Notify adapter decisions intact. A core default must never
override one of those explicit boundaries.

`CPP_RUNTIME_INTERNAL` is not a Host capability. It carries
`cpp_host_binding = NOT_APPLICABLE`, does not grant filesystem, process,
network, device, or native-handle authority to scripts, and does not require a
script parity ID. The native subsystem that replaces the Python source remains
responsible for its own unit, integration, and platform tests.

The source split is applied per occurrence before aggregation. Therefore the
same resolved symbol can produce a `CPP_RUNTIME_INTERNAL` decision when called
inside `core/*` and a `HOST_BINDING_REQUIRED` or
`SCRIPT_LANGUAGE_OR_MODULE` decision when called from `module/*`. Ownership is
not inferred from a receiver suffix or final identifier segment.

## Consequences

- All 15,469 observed Python operation sites now have a disposition. Strict
  generation succeeds with zero unresolved dispositions, zero Host contract
  gaps, and zero parse errors.
- Completing the operation inventory does not claim that C++ runtime
  implementations, Host adapters, Python parity, service end-to-end behavior,
  or platform smoke tests are complete.
- `core/*` remains evidence for native behavior and parity design; it is not
  accepted as hot-update script source.
- Calls from migrated modules into device, vision, OCR, configuration, logging,
  scheduling, resource, and service capabilities remain governed by the Host
  catalog and capability checks.

## Verification

Migration fixtures exercise identical unknown calls in `core/*` and
`module/*`, exact privileged-rule precedence, deterministic decision IDs, and
strict generation. The checked-in operation JSON and generated matrix are
regenerated byte-for-byte from the pinned `baas-dev` revision. Documentation
tests validate the v5 source scopes, the `CPP_RUNTIME_INTERNAL` disposition,
all remaining Host mappings, and this ADR.
