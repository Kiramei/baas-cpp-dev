# Python Operation Index Audit

This is Phase 0 evidence, not a parity or completion claim. The operation index
was generated by statically parsing the `baas-dev` snapshot; no inspected Python
module was imported or executed.

## Reproduction

From a `baas-cpp-dev` checkout with `baas-dev` as a sibling directory:

```powershell
python scripts\migration\operation_index.py `
  --python-repo ..\baas-dev `
  --output docs\script-runtime\evidence\operation-index.json `
  --matrix docs\script-runtime\MIGRATION_MATRIX.md `
  --evidence-link evidence/operation-index.json

python scripts\migration\operation_index.py `
  --python-repo ..\baas-dev `
  --output build\migration-operation-index-strict.json `
  --strict
```

The first command returned 0. The strict command returned 1 because unresolved
classification work remains. Two consecutive reports, including the strict
report, were byte-identical.

## Unclassified evidence and Phase 0 gate

All 4,022 unclassified IDs are listed in `unclassified_operation_ids` and have
full entries in the JSON report. High-frequency examples include GUI
translation and widget calls, activity helper dispatch, `super()` lifecycle
calls, clock/process/network calls, internal module helpers, and 296 dynamic or
chained expressions. These deliberately remain `UNCLASSIFIED` rather than
being assigned to a plausible-looking owner or binding without evidence.

Phase 0 therefore remains incomplete: strict classification is non-zero, and
the roadmap separately still requires representative golden traces,
performance/package baselines, and platform/emulator smoke prerequisites. The
generated matrix is an authoritative work queue, not proof that host bindings
or parity tests already exist.
