# Python representative workflow golden traces

Status: verified Phase 0 baseline evidence; Python-to-C++ parity remains pending

The deterministic corpus is checked in as
`evidence/python-golden-traces.json`. It captures four representative offline
Python workflows at real host boundaries and is intended as a future C++ replay
oracle. Capturing the Python side does not claim that a C++ binding exists or
passes parity.

## Source authority

The corpus locks two related `baas-dev` revisions:

- production source: `75bbacb545bc87e9510d85cbe8034f9180397004`;
- opt-in trace source: `3a8f58585b69bf7cf54fe66115352b41f4094aa3`.

The trace revision has exactly the production revision as its parent. Its one
commit adds `core/parity_trace.py`, explicit `Baas_thread` trace injection,
tests, and documentation; the default production path remains disabled. The
generator refuses a different revision, a different parent, or a dirty trace
checkout. It records both full commits and the clean/parent verification flags.

## Executed workflows

| Workflow id | Actual Python entrypoint and boundary | Deterministic substitution |
| --- | --- | --- |
| `configuration.snapshot_patch` | `service.conf.manager.ConfigManager.apply_patch`; reads a temporary config, returns a snapshot, applies JSON Patch, persists JSON, and publishes a sync update | temporary root, fixed file/service timestamp, fixed fixture config; a credential sentinel verifies recursive redaction |
| `image.cafe_match` | `service.injection._patch_cafe_reward` installs the production service implementation, then `module.cafe_reward.match` runs OpenCV template matching against a real tracked template | synthetic 1280x720 image containing the real template; pixels are hash-only `ImageFixture` values |
| `scheduler.queue_rebuild` | `core.scheduler.Scheduler.update_valid_task_queue` and `heartbeat` read event JSON, filter/order tasks, emit the selected/remaining queue | temporary event file, fixed wall clock and epoch |
| `orchestration.grid_actions` | `module.explore_tasks.task_utils.run_task_action` dispatches real grid actions through the trace-enabled `Baas_thread.click` facade | formation/pop-up functions and the physical click implementation are deterministic host doubles; dispatcher and trace facade are production code |

The first three boundaries exercise configuration/filesystem, image matching,
and scheduling. The fourth additionally freezes operation-dispatch ordering and
the three click calls generated by `click` plus `choose_and_change`.

The evidence contains 30 parity records across the four sessions. Current
high-level fixture assertions are:

- configuration retains an unknown field and persists difficulty `3`;
- image matching finds the embedded tracked template;
- scheduler selects `task_a` before `task_b`;
- grid orchestration makes three traced physical click calls.

These assertions are validated by the standard-library test suite; they are not
inferred from source names alone.

## Determinism, schema, and limits

`scripts/migration/capture_python_golden_traces.py` is a standard-library
controller. It launches one isolated Python 3.11 child with the audited
`baas-dev` dependencies. The child uses:

- stable workflow/session/task ids;
- injected step clocks and fixed synthetic wall time;
- an explicitly marked no-RNG fixture source;
- sorted compact parity records and sorted pretty outer JSON;
- temporary directories whose absolute paths are never serialized;
- `SafeSerializer` redaction and hash-only image/binary values.

The strict outer schema is `baas.python-golden-traces` version 1. It requires
the exact v1 workflow order and categories, contiguous trace sequence numbers,
one session start/end pair, matched begin/terminal operation ids, zero
error/cancel records, and zero dropped/write-error events. Unknown or missing
root/workflow keys fail validation.

Hard bounds are recorded in the artifact and enforced during generation:

| Limit | Value |
| --- | ---: |
| workflows | 8 maximum; v1 has 4 |
| records per workflow | 64 |
| encoded bytes per record | 65,536 |
| complete evidence | 262,144 bytes |

Every workflow has a SHA-256 over its compact canonical record array. The
summary has a SHA-256 over the complete ordered workflow array. The checked-in
pretty JSON file is 26,686 bytes with SHA-256
`a3bf59d07d8b31863d3c7ed95642cd9e1d13171f3f29036eddff93a1d0e9e695`.

The artifact has no generation timestamp or absolute user path. Synthetic
`wall_time_ns`, `monotonic_ns`, config timestamps, and durations are fixture
values needed for replay, not observations of the capture machine.

## Rebuild and check

From the normal `baas-cpp-dev` checkout, with the read-only trace worktree and
the production `baas-dev` virtual environment already present:

```powershell
python scripts\migration\capture_python_golden_traces.py `
  --python-repo ..\.worktrees\baas-dev-parity-trace `
  --python-executable ..\baas-dev\.venv\Scripts\python.exe `
  --output docs\script-runtime\evidence\python-golden-traces.json

python scripts\migration\capture_python_golden_traces.py `
  --python-repo ..\.worktrees\baas-dev-parity-trace `
  --python-executable ..\baas-dev\.venv\Scripts\python.exe `
  --output docs\script-runtime\evidence\python-golden-traces.json `
  --check
```

`--check` rebuilds into memory and never writes the committed file. A mismatch
returns 1 and reports the first JSON Pointer plus committed/rebuilt file hashes.
A source, import, workflow, timeout, schema, bound, or sensitive-data failure
also returns 1 with a compact diagnostic. Invalid CLI setup returns 2.

The default Python search first checks the trace checkout's `.venv`, then the
normal sibling `baas-dev/.venv`. `--python-executable` is preferred for an
unambiguous reproduction. The checked-in runtime versions are Python 3.11.9,
OpenCV 4.8.1, and NumPy 1.26.4; changing them is an intentional corpus update
that must be reviewed with the rebuilt trace differences.

CI runs the standard-library schema/hash/tamper/CLI tests without requiring a
second repository. A true source rebuild additionally requires the pinned
read-only `baas-dev` trace checkout and its dependencies, so it is an explicit
local/release evidence command rather than an implicit network checkout.

## Excluded paths and completion boundary

The capture does not execute device, OCR, service, or Tauri paths. Specifically
it does not run:

- ADB, scrcpy, Nemu, uiautomator, emulator, screenshot-device, or input-device
  I/O;
- OCR initialization or inference;
- network access, repository update checks, or authentication;
- FastAPI lifespan, HTTP, WebSocket, named-pipe, or Unix-socket transport;
- Tauri launch, Android foreground/JNI lifecycle, or packaged application.

Those paths remain pending in their owning roadmap phases and must not be
inferred from this corpus. The Phase 0 item completed here is only the capture
of representative Python golden traces. Phase 0 as a whole remains incomplete
while operation classification and other exit evidence remain open, and no
Python-to-C++ parity result exists until a C++ replay implementation compares
against these records.
